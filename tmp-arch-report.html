Candidate #1

Duplicate CRUD Sections — Admin Panel

Strong

Files

components/admin/sections/PostsSection.tsx EventsSection.tsx GallerySection.tsx MathetesSection.tsx OfficeBearersSection.tsx LivingRoomSection.tsx

Problem

All 6 admin CRUD sections follow an identical pattern — each is a 200–400 line client component with duplicated modal forms, state management (form, editingId, saving, error), input handling, and save/cancel logic. The useAdminCrudSection hook extracts list fetching and deletion, but each section still defines its own form state, validation, file upload wiring, and modal layout. A new content type requires copying an entire section and changing field names. This is the textbook definition of a shallow module — the interface (props, state shape) is nearly as complex as the implementation.

Solution

Extract a generic CRUD module behind a declarative interface. Each section becomes a data declaration — a schema object that specifies fields, types, validation rules, and API endpoints — rather than a component with imperative state management. The generic module handles list rendering, modal lifecycle, form state, save/delete, and error handling.

Benefits

  • Locality: Adding a new content type = 1 schema declaration (≈30 lines) instead of 1 new component (≈300 lines)
  • Deletion test: Removing a schema would concentrate all CRUD logic in one place — exactly where it should be
  • Test surface: The generic module's interface (the schema) becomes the single test surface for CRUD behavior
  • Leverage: A bug fix in form validation applies to all 6 sections at once

Candidate #2

Type Duplication Across the Codebase

Strong

Files

app/page.tsx (line 35: Event, 44: Post, 96: Person) app/events/page.tsx app/gallery/page.tsx (line 31: Photo) app/office-bearers/page.tsx (line 16: Person, 26: Team) app/mathetes/page.tsx (line 19: MathetesEntry) app/cezo-mepu/page.tsx (line 18: Location, 28: Supervisor) app/living-room/page.tsx (line 13: Episode) app/developers/page.tsx (line 17: Developer)

Problem

The same database-backed types are re-defined in multiple files — Person and Team appear in 3 files each, Event in 2 files, Post in 2 files, and Photo/MathetesEntry/Episode are duplicated between page components and admin sections. These are exact mirrors of the Supabase table schemas. Changing a column name means updating 2–5 files. There's no single source of truth.

Solution

Create a lib/types/ module that re-exports all database-backed types inferred from Supabase schema (or manually defined). Each page/component imports from the shared module. This is a one-time refactor with immediate payoff: type changes propagate everywhere.

Benefits

  • Locality: All type definitions live in one place — you understand the domain model from one directory
  • Deletion test: Deleting a type would just remove its export; no hunt for copies
  • Levrage: Adding a Supabase table = one type definition, used everywhere

Candidate #3

Three Supabase Clients — Inconsistent Usage

Worth exploring

Files

lib/supabase.ts lib/supabase-browser.ts lib/supabase-server.ts app/living-room/page.tsx (uses createClient directly)

Problem

The project defines 3 Supabase client factories for different contexts, but the living-room/page.tsx bypasses all of them and calls createClient from @supabase/supabase-js directly. Meanwhile, 10 other server components import lib/supabase.ts, and app/developers/page.tsx uses lib/supabase-server.ts for auth-aware queries. The inconsistency means there are now 4 ways to create a Supabase client, increasing cognitive load and making it unclear which factory to use in what context.

Solution

Consolidate the living-room/page.tsx to use the existing lib/supabase.ts client (which is the correct one for server components reading public data). Consider whether the 3-client pattern can be simplified — perhaps a single factory with a context flag.

Candidate #4

Hardcoded Data in Shared Components

Worth exploring

Files

components/HeroSlider.tsx components/Navbar.tsx components/Footer.tsx

Problem

HeroSlider hardcodes 4 image URLs, Navbar hardcodes 7 navigation links, and Footer hardcodes 3 social media URLs. These are all content concerns embedded in component code. Changing a hero image requires a code deploy; adding a nav item requires editing the component. This creates coupling between UI structure and content — a classic shallowness signal.

Solution

Extract hardcoded data into configuration files (e.g., lib/navigation.ts, lib/social-links.ts, lib/hero-images.ts) or into Supabase tables for admin-editable content. This separates content from presentation.

Candidate #6

Missing CONTEXT.md — Domain Vocabulary Gaps

Worth exploring

Files

CONTEXT.md (does not exist) docs/ (architecture.md, design.md, memory.md, PRD.md)

Problem

There is no CONTEXT.md file to serve as the canonical domain glossary. Currently, domain vocabulary is scattered across docs/architecture.md, docs/design.md, and docs/memory.md. Key terms like "Cezo Mepu", "Mathetes", "The Living Room", and the relationship between "Office Bearers", "Teams", and "Youth Supervisors" are implicitly understood but nowhere explicitly defined in one place.

Solution

Create a CONTEXT.md file that defines the domain model in plain language — what each content type represents, the organizational hierarchy, and relationships between entities. This serves as the shared vocabulary for both human developers and AI agents.

Candidate #7

API Route Pattern — Generic CRUD Handler

Worth exploring

Files

app/api/admin/posts/route.ts app/api/admin/events/route.ts app/api/admin/gallery/route.ts app/api/admin/mathetes/route.ts app/api/admin/office-bearers/route.ts app/api/admin/living-room/route.ts

Problem

Each admin API route follows the same pattern — a POST handler that switches on an action field (create/update/delete) and performs the corresponding Supabase operation. This is the server-side mirror of the admin section duplication. Each route re-implements auth checks, error handling, and the action dispatcher pattern.

Solution

Create a generic admin API handler that accepts the table name, allowed actions, and any custom business logic as callbacks. Each entity route becomes a thin declaration of permissions and table mappings.

Top Recommendation

Start with #1: Generic CRUD Section + #2: Shared Types

These two candidates are mutually reinforcing. A generic CRUD module needs shared type definitions, and centralizing types makes the generic module type-safe. Together they eliminate ~1,500 lines of duplicated code (roughly 40% of the admin panel) and establish the type foundation that all other architectural improvements depend on. The refactor is purely mechanical — no business logic changes — and can be done incrementally, one section at a time.
